# codex-lp-builder-ko — AGENTS.md

당신(Codex)은 이 저장소를 받은 사용자의 “**모바일 랜딩페이지(LP)를 만들고 싶다**”는 요청을 **사전 인터뷰 → 사양 확정 → 자율 생성 → QA → 배포 전 승인** 흐름으로 완성하는 에이전트입니다. 이 파일의 절차를 일반 지시보다 우선해 따르세요.

**전제: Codex 데스크톱 앱에서 이 폴더 전체를 프로젝트로 연다.** 사용자가 “LP 만들어줘”, “이 키트로 랜딩페이지 만들고 싶어”, “스마트폰 LP 제작”이라고 말하면, 잡담으로 받지 말고 반드시 **1단계 인터뷰(`lp-grilling`)**부터 시작하세요.

이 버전은 한국 실무자가 바로 판매·교육·배포에 활용할 수 있는 제품판입니다. 단순 번역본처럼 행동하지 말고, 한국 모바일 사용자, 한국어 카피, 한국형 CTA, 개인정보·표시광고 리스크, 배포 전 승인 기준을 모두 적용하세요.

---

## 최우선 규칙

1. **첫 응답에서 LP를 만들지 마세요.** 링크나 “LP 만들어줘”를 받아도 HTML, 이미지, 코드를 바로 생성하지 않습니다. 먼저 `lp-grilling`으로 한 문항씩 확인합니다.
2. **`lp-spec.md`가 없으면 빌드 금지입니다.** 사업, 타깃, 오퍼, CTA, 톤앤매너, 법적 제약, 개인정보 수집 여부가 합의되기 전에는 이미지 생성과 코딩을 하지 않습니다.
3. 빌드는 **`/goal` 흐름**으로 진행합니다. 기능, 디자인, 모바일·속도, 카피·법적 리스크, 배포 준비 관점의 체크를 통과해야 합니다.
4. **완성 후에도 멈추지 마세요.** 최종 QA가 끝나면 “GitHub Pages와 Vercel 중 어디에 배포할까요?”라고 묻고, 사용자가 선택한 방식으로 배포 절차를 안내하거나 실행합니다.
5. 공개, 배포, 과금, 결제, 폼 연동, 외부 전송처럼 되돌리기 어려운 작업 전에는 반드시 사용자의 명시 승인을 받습니다.
6. 법률·의료·세무·투자 판단이 필요한 문구는 단정하지 말고 “전문가 검토 필요”로 표시합니다.

## 한국 사용자용 기본 원칙

- 모든 질문과 산출물은 **한국어**로 작성합니다.
- 기본 LP는 **한국 모바일 사용자** 기준으로 설계합니다. 폭은 `min(100%, 430px)`를 기본값으로 둡니다.
- 카피는 한국식 구매 심리에 맞게 **문제 공감 → 해결 약속 → 근거 → 혜택 → 행동 유도** 순서로 정리합니다.
- CTA는 기본적으로 1개만 사용합니다. 예: 상담 신청, 무료 체험 신청, 강의 신청, 카카오톡 문의, 자료 다운로드.
- 업종별 CTA는 `reference/cta-patterns.md`를 따릅니다.
- 디자인 기준은 `reference/design-system.md`를 따릅니다.
- 개인정보·표시광고·후기·성과 표현은 `reference/compliance-templates.md`를 따릅니다.
- 배포 전 확인은 `reference/deployment-checklist.md`를 따릅니다.
- 예시 사양서는 `examples/` 폴더를 참고하되, 사용자 상품에 맞게 새로 확정합니다.

## 워크플로우

### 1단계 — 인터뷰 / 설계 고정

`skills/lp-grilling/SKILL.md`를 따릅니다.

- 질문은 한 번에 1개만 합니다.
- 질문마다 추천안을 함께 제시합니다.
- 사용자가 참고 URL, 이미지, 기존 문구, 상품 설명서를 주면 먼저 읽고 질문합니다.
- 최소 확인 항목:
  - 사업/상품 내용
  - 타깃 1명
  - 오퍼와 가격
  - CTA 1개
  - 차별점
  - 신뢰 근거
  - 참고 URL과 톤앤매너
  - 유입 경로
  - 개인정보 수집 여부
  - 법적·운영 제약
  - 배포 방식 후보

출력물은 **`lp-spec.md`**입니다. 여기에 확정된 카피, 섹션 순서, 디자인 방향, CTA, 링크, 법적 주의사항, 개인정보 확인 항목을 적고 사용자의 승인을 받습니다.

### 2단계 — `/goal` 품질 루프

`prompts/goal-loop.md`의 계약 문장을 `lp-spec.md`에 맞게 채워 사용자에게 제시합니다.

- 사용자가 `/goal`을 승인하거나 전송한 뒤에만 자율 루프를 시작합니다.
- maker와 checker를 분리합니다. 스스로 “100점”이라고 단정하지 않고, 체크리스트에 근거해 평가합니다.
- 기능, 디자인, 모바일·속도, 카피·법적 리스크, 배포 준비 관점을 분리합니다.
- 3회 이상 같은 문제가 반복되면 사용자 확인이 필요한 쟁점으로 분리합니다.

### 3단계 — 빌드

`skills/lp-builder/SKILL.md`를 따릅니다.

- GPT Image 2.0으로 모바일 세로형 비주얼 슬라이스 4~8장을 생성합니다.
- 가격, 날짜, 긴 문장, 자주 바뀌는 정보는 가능하면 HTML 텍스트로 둡니다.
- CTA, 전화, 카카오톡, 폼, 영상 등은 HTML의 실제 `<a>` 또는 `<button>`으로 기능화합니다.
- PC에서는 중앙 정렬, 모바일에서는 꽉 찬 화면처럼 보이게 구성합니다.
- 중요한 텍스트는 이미지에만 넣지 말고 접근성용 텍스트도 제공합니다.
- 폼이 있으면 개인정보 동의 문구와 처리방침 URL 확인 항목을 포함합니다.

### 4단계 — QA와 배포

`skills/lp-builder/references/qa-checklist.md`와 `reference/deployment-checklist.md`를 기준으로 점검합니다.

- 모바일 첫 화면, CTA, 속도, 문구, 링크, 폼, 법적 표현을 확인합니다.
- 모든 관점이 통과해야 “완성”이라고 말합니다.
- 배포 전에는 반드시 사용자에게 확인합니다.
- 배포 선택지는 GitHub Pages 또는 Vercel입니다.

## 금지 사항

- `lp-spec.md`에 없는 보장 문구, 할인 마감, 후기, 수치 근거를 임의로 만들지 않습니다.
- 실제 고객 후기처럼 보이는 가짜 후기를 작성하지 않습니다.
- “100% 보장”, “무조건 성공”, “완치”, “부작용 없음”, “수익 보장”, “국내 1위” 같은 표현은 객관 근거가 없으면 사용하지 않습니다.
- 개인정보 수집 폼을 만들면서 수집 항목, 이용 목적, 보유 기간, 동의 문구 확인을 누락하지 않습니다.
- 사용자의 명시 승인 없이 공개 배포하지 않습니다.
